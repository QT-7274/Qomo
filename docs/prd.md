# Product Requirements Document - Qomo

**Author:** drogbaqu
**Date:** 2025-12-28

## Executive Summary

Qomo 是一个面向高频使用对话式 AI 的用户的可视化 Prompt 资产库与组装工作台。它将 Prompt 从“每次重写的文本”升级为“可复用的资产”：用户以**模板（Template）**为主进行复用与产出，并在提问前通过“选择模板 → 填变量 → 选择约束包 → 生成预览 → 一键复制/导出”的主路径，快速生成结构清晰、约束明确、可直接投入外部 AI 的最终 Prompt（一期不做模型直连）。

Qomo 解决的问题是：高频用户在每次提问前反复拼凑角色/目标/边界/输出格式等要素，过程繁琐且容易遗漏关键约束，导致输出质量波动、返工增多；同时优质 Prompt 分散在聊天记录/笔记/剪贴板中，难以结构化沉淀、复用与迭代。

### What Makes This Special

- **模板驱动主路径**：默认只需要“选模板 + 填变量”即可完成导出；模板入口支持最近使用/收藏/搜索，变量以表单方式展示并对未填项高亮提示。
- **模块化能力（进阶）**：模块（Module）作为可复用片段用于增强模板的可维护性与可组合性，但不强迫新手理解模块；主要露出在模板管理/专家模式中。
- **约束包产品化**：把输出格式、边界与质量检查等“有效约束”做成可组合选项，按场景复用，降低试错成本。
- **导出前轻量校验**：对占位符未填、输出格式一致性、疑似敏感信息提供非阻断提示与遮罩建议（仅对空内容/全占位符做硬拦截），目标是“导出后少返工”。
- **安全与可控边界**：一期不直连模型；资产与单次问题内容分离，**单次问题默认不入库**，仅在用户显式选择“保存为资产”时才沉淀。

### Non-Goals (MVP)

- 不做模型直连/调用与对话托管
- 不做社区分享、评分与衍生
- 不做团队协作与权限系统
- 不做复杂版本分支/对比/回滚（先保证可迁移导入导出）

## Project Classification

**Technical Type:** web_app
**Domain:** general
**Complexity:** low（领域合规复杂度低；产品交互与资产模型复杂度中等）
**Project Context:** Greenfield - new project

该产品更符合浏览器端工作台式应用形态，核心挑战集中在资产模型、组装工作流、校验与迁移体验，而非强监管行业的合规与审计要求。

## Success Criteria

### User Success

- **更快完成可用 Prompt**
  - 用户可通过“模板驱动主路径”快速得到可用的最终 Prompt（选择模板 → 填变量 → 约束包 → 预览 → 复制/导出）。
- **导出后少返工**
  - 导出前轻量校验能减少“导出后立刻反复再生成/再导出/再编辑”的情况（作为质量代理指标）。
- **复用变简单**
  - 用户能在后续会话中稳定复用：模板复用、模块复用、历史组合复用（至少其一，并实际参与生成）。

### Business Success

- **可验收阈值（暂定）**：以“用户使用产品的次数”作为首版验收口径
  - 默认统计窗口：**按周（Weekly）**
  - “使用一次”必须是可被系统客观记录的价值事件，而不是仅打开页面或停留浏览。

#### 指标之间的关系（确保自洽）

- **North Star（北极星）：WAEP（Weekly Active Exporters with Reuse）**
  - 定义：每周至少一次 `export_success/copy_success`，且该次会话 `reuse_used=true`（模板/模块/历史组合至少其一并实际参与生成）。
- **Weekly Usage（周使用次数）**
  - 定义：每周 `export_success/copy_success` 的有效计数（用于衡量产出强度）。
- 解释：
  - Weekly Usage 反映“产出强度”
  - WAEP 反映“产出价值 + 复用沉淀”（防止只追次数）

#### 统计窗口与口径约定（用于验收）

- **Weekly（按周）**：周的边界以用户本地时区为准（客户端记录 `tz_offset`，报表按用户本地周汇总），周起始日为周一（Monday）。

#### 离线与无登录条件下的计量原则

- 本期不做登录/同步，但为支持指标验收，可使用**匿名用户标识**（本地生成并持久化）进行去重与周统计。
- 遥测遵循最小化原则：仅采集事件与必要属性（不上传完整 Prompt 原文；如需去重可使用内容哈希/长度等摘要信息）。

#### 事件口径（把“去重/硬拦截/系统失败/权限拒绝”闭环为验收口径）

- `export_success/copy_success`（计入 Weekly Usage）
  - 前置条件：最终 Prompt **非空**且**不为全占位符**
  - 成功标准：系统侧完成复制/导出动作，无异常
  - 去重口径：同一用户在 30 秒内对同一份**规范化文本**（trim/统一换行等）计算得到的内容哈希重复导出，仅计 1 次；其余记为 `export_duplicate`，不计入 Weekly Usage
- `validation_blocked`（唯一硬拦截）
  - 触发条件：空内容/全占位符
  - 说明：这不属于“系统失败”，属于“业务拒绝”，不应计入失败率
- `export_error/copy_error`（系统失败，约束对象）
  - 触发条件：系统异常导致导出/复制未完成
  - 说明：必须可观测、可定位、可回归；属于“导出零失败”的约束对象
- `export_denied`（权限/策略拒绝）
  - 触发条件：浏览器权限或安全策略拒绝（如剪贴板权限被拒）
  - 说明：不算系统 bug，但必须提供兜底路径以完成导出（例如下载为 `.txt/.md` 或提供可全选文本）

#### 去重与哈希规范（避免“刷次数/误计数”）

- `dedup_window_seconds = 30`
- 内容哈希基于规范化文本（trim/统一换行等）计算，避免同内容因格式差异重复计数。

#### 质量护栏（避免指标跑偏）

- **Reuse Rate（复用参与率）**：复用参与生成且导出的会话占比。
- **Valid Export Proxy（少返工代理指标）**：导出后短窗口内的高频重复生成/导出/编辑比例（作为质量门槛/护栏）。

### Technical Success

- **离线优先、免登录（本期不做登录/同步）**
  - 模板、模块、约束包、最近组合、历史组合均需本地持久化。
  - 提供可迁移能力：导入/导出（例如 JSON）用于备份与跨设备迁移（即使不做账号同步）。
- **导出/复制“系统侧零失败”**
  - **验收目标**：系统侧 `export_error/copy_error` 目标为 0（不接受系统原因导致的导出/复制失败）。
  - **SLO 报表阈值**：按周统计失败率 \(\le 0.01\%\)（超出视为回归/阻断发布）。
  - 唯一硬拦截：空内容/全占位符；其余校验均为非阻断提示。
  - 权限/策略拒绝需兜底（`export_denied` 不应让用户无法完成导出）。
- **性能与稳定性门槛（体验可测，默认目标）**
  - 生成到预览可用：p95 \(\le 300\)ms，p99 \(\le 800\)ms
  - 打开模板/切换约束包：p95 \(\le 200\)ms
  - 复制/导出动作完成：p95 \(\le 200\)ms
  - 前端稳定性：JS 错误、白屏等需可观测并作为质量门槛。

### Measurable Outcomes

#### 核心事件（建议埋点最小集）

- `generate_success`
- `export_success` / `copy_success`（Weekly Usage 的计数来源）
- `reuse_used`（模板/模块/历史组合至少其一参与生成）
- `validation_warning_shown`（非阻断：未填变量/敏感信息/格式一致性）
- `validation_blocked`（硬拦截：空内容/全占位符）
- `export_duplicate`（重复导出，不计入 Weekly Usage）
- `export_error/copy_error`（系统失败，目标为 0）
- `export_denied`（权限/策略拒绝，需兜底）

#### 关键指标（用于验收与迭代）

- **Activation-1**：首次会话完成 `export_success` 或 `copy_success`
- **Activation-2**：后续会话 `reuse_used=true` 且完成 `export_success/copy_success`
- **Weekly Usage**：每周有效导出次数（按上文定义）
- **WAEP**：每周至少一次导出且复用参与（按上文定义）
- **Reuse Rate**：`sessions(reuse_used=true AND export_success)/sessions(export_success)`
- **Valid Export Proxy（少返工代理，质量护栏）**：导出后短窗口内的高频重复生成/导出/编辑比例（作为质量门槛/护栏）

## Product Scope

### MVP - Minimum Viable Product

- **模板驱动主路径（默认）**
  - 选择模板 → 填变量（未填高亮/提示）→ 选择约束包 → 生成预览 → 复制/导出
- **模板与模块（两条都完整可用，但主路径不复杂化）**
  - 模板：创建/编辑/删除、标签/搜索/最近使用、变量槽位管理
  - 模块：创建/编辑/删除、标签/搜索、可复用片段
  - 组装：模板可引用模块；提供专家模式支持直接模块拼装（不强迫新手进入）
- **约束包（内置少量，克制范围）**
  - 输出约束、边界约束、质量约束三类，作为可组合选项
- **导出前轻量校验**
  - 未填变量提示（默认非阻断）
  - 输出格式一致性提示（默认非阻断）
  - 疑似敏感信息提示/遮罩建议（默认非阻断）
  - 空内容/全占位符硬拦截
- **离线优先（无登录）**
  - 本地持久化 + 导入/导出用于迁移/备份

### Growth Features (Post-MVP)

- 账号体系与多端同步（可选）
- 导入资产可信机制增强（来源/版本/变更对比、更强提示）
- 更细粒度版本管理（对比、回滚等）

### Vision (Future)

- 社区/市场：模板与模块的发布、评分、衍生与版本演进
- 团队协作：团队资产库治理、权限与审计（如需要）

## User Journeys

**Journey 1: 林然（产品经理）- 30 分钟交付一份可用的 PRD Prompt**
林然周一早上要写本周第一份 PRD，并希望在午前把初稿交给团队。她打开 Qomo，不想“研究工具”，只想尽快拿到一份结构清晰、能直接丢给外部 AI 的最终 Prompt。她在首页看到“最近使用/收藏/搜索”，直接选中“PRD 初稿生成（模板）”。接着她只需要填写 1-2 个变量：产品名称、目标用户、这次迭代的边界；未填项会被高亮，并可一键跳转定位。
系统为她推荐一组默认约束包（输出格式 + 边界 + 质量），高级选项折叠显示，避免她纠结。点击生成后，她看到预览已经是可读的结构化 Prompt。导出前，系统提示“有 1 个变量未填，但可继续导出”，并解释这是非阻断提醒。林然直接复制导出，把 Prompt 粘贴到外部 AI 后，第一轮输出就满足结构要求，她只需要补充少量背景即可提交初稿。她明显感觉“导出后少返工”，并把这个模板加入收藏，准备周报也复用。

**Journey 2: 林然（产品经理）- 边界场景：复制受限也能完成导出**
在公司安全策略较严格的浏览器环境里，林然点击“复制”时遇到剪贴板权限拒绝。Qomo 明确提示“当前环境不允许写入剪贴板”，并弹出统一的“导出动作面板”，提供至少两条可完成导出的路径：
1) 下载为 `.md/.txt` 文件
2) 展示可全选的最终文本区（提供快捷键提示）
林然仍然在同一会话中完成导出动作，没有被迫反复重试或丢失编辑状态。对她来说，价值链路没有断裂。

**Journey 3: 周骁（开发者）- 为多项目维护一套可复用模块库**
周骁在不同项目里经常需要类似的提示词结构：代码评审、技术方案、测试用例、排障总结。他不希望每次从零拼 Prompt，也不想维护一堆难以理解的长模板。他在 Qomo 的模板管理页进入专家模式，维护一套模块库：角色（Reviewer）、质量门槛（不给结论先列假设与缺失信息）、输出格式（按小标题/表格/JSON）、安全边界（不泄露敏感信息）等。每次写新模板时，他把模块组合进模板骨架，再加少量变量槽位（如项目名、变更摘要、目标读者）。
当某个模块升级（例如加入更严格的“只基于给定上下文”），周骁希望能在多个模板里复用该更新而不产生维护灾难。他也会频繁导出/导入这套资产：一份给工作电脑，一份给个人电脑，确保迁移和备份简单可控。对他来说，Qomo 的价值在于“模块化资产可维护 + 可迁移 + 组合后能稳定导出”。

**Journey 4: 迁移/导入导出型用户 - 从旧资料搬运资产并完成清理**
这位用户手里已经有大量 Prompt：散落在笔记、文档、聊天记录里。TA 想把“可复用的部分”沉淀成模块/模板，但担心导入后乱成一团、变量不一致、甚至带入不安全的隐藏指令。TA 使用 Qomo 的导入功能批量导入（例如 JSON 或粘贴文本识别），系统在导入过程中做最小但高价值的校验：
- 识别变量占位符与命名冲突
- 提示可能的敏感信息（并建议遮罩）
- 提示可疑指令（例如“忽略以上规则”等），以“摘要提醒 + 可展开查看命中片段”的方式呈现
导入不默认阻断，但允许 TA 选择“继续导入并打标（需复查）”。导入完成后，TA 在管理页进行清理：合并重复模块、统一变量命名、为常用模板补齐约束包默认选项。最后，TA 导出一份“干净的资产包”作为备份与跨设备迁移。

**Journey 5: 隐私敏感用户 - 透明度与可控优先**
这位用户非常在意：Prompt 可能包含敏感信息，且不希望自己的内容被上传或用于训练。TA 打开 Qomo 后，首先看到清晰的承诺与解释：单次问题默认不入库；遥测仅采集最小化事件；不会上传完整 Prompt 原文。TA 在设置中能一键关闭遥测、重置匿名用户标识、以及查看“会被记录的事件列表”。在导出前如果命中疑似敏感信息提示，系统以非阻断方式提醒并提供遮罩建议。TA 的心理预期是“默认安全 + 明确透明 + 可随时关掉”，这决定了 TA 是否会长期使用与沉淀资产。

**Journey 6: 本地数据丢失/恢复 - 清缓存/误删后仍可找回**
某天用户清理浏览器缓存或误删了本地数据，打开 Qomo 发现模板/模块不见了。系统不把这当作“用户自己活该”，而是提供可理解的恢复路径：
- 在设置页清晰提示本地存储机制与风险（尤其是无登录条件下）
- 提供“导入资产包”恢复流程，并在导入时允许选择“合并/覆盖/新建副本”
- 对导入文件提供版本信息与兼容提示，避免因版本不匹配导致导入失败
这条旅程的目标是：让“离线优先”可被信任，不因一次意外就彻底放弃产品。

### Journey Requirements Summary

这些旅程揭示了对 Qomo 的关键能力需求：
- **主路径能力（模板驱动）**：模板选择（最近/收藏/搜索）→ 变量填写（少量必填、未填高亮/跳转）→ 约束包推荐（默认推荐 + 高级折叠）→ 预览 → 复制/导出
- **可靠导出与兜底**：剪贴板权限拒绝/不可用时的导出兜底（下载 `.md/.txt`、可全选文本区），确保用户仍可完成导出
- **复用与资产治理**：模块与模板的创建/编辑/搜索/标签化；跨模板复用模块以降低维护成本；专家模式入口
- **迁移能力**：导入/导出资产包用于备份与跨设备迁移；资产包版本号与向后兼容提示
- **导入安全提示**：对导入内容进行可疑指令/敏感信息的最小提示（摘要 + 可展开命中片段 + 打标复查）
- **隐私与透明度**：遥测最小化、设置中可关闭遥测/重置匿名标识/查看事件清单
- **恢复与覆盖策略**：本地数据丢失后的导入恢复路径，以及合并/覆盖/副本策略与状态保持

## Innovation & Novel Patterns

### Detected Innovation Areas

**Innovation 1: 约束包（Constraint Packs）产品化（并嵌入模板驱动主路径）**  
Qomo 的核心创新点是将“有效约束”（输出格式、边界约束、质量检查等）从一次性 Prompt 经验中抽离，做成可组合的约束包，并在模板驱动主路径中以低摩擦方式应用（默认推荐 + 高级折叠）。这使用户在不学习复杂 Prompt 工程的前提下，也能稳定获得结构化、边界清晰、少返工的可用 Prompt；高阶用户则可以维护与复用约束包作为长期资产。

### Market Context & Competitive Landscape

该创新并非宣称“从未有人做过”，而是以新的组合方式解决“提问前一公里”的质量与稳定性问题：
- 传统笔记/文档/剪贴板：能存放但难以资产化组装，缺少校验与质量护栏
- Prompt 库/收藏：偏发现与管理，缺少生成工作流与导出前校验
- 纯模板：可复用但难治理与演进，约束无法产品化沉淀

Qomo 通过“模板主路径 + 约束包可复用 + 非阻断校验 + 数据闭环（少返工代理指标）”形成差异化。

### Validation Approach

**目标：证明“有约束包”能显著降低导出后的返工行为，并提升可用导出质量。**

- **实验设计（A/B 或准实验）**
  - A（控制组）：无约束包（或不提供结构化约束包选择）
  - B（实验组）：有约束包（默认推荐 + 可选组合）
- **分流与统计**
  - 按匿名用户标识分桶（避免同一用户跨组污染）
  - 核心观察窗口：导出后短窗口 \(N = 5\mathrm{min}\)（沿用少返工代理指标口径）
- **核心观测指标**
  - 少返工代理指标（Valid Export Proxy）：导出后 \(N\) 分钟内的高频重复生成/导出/编辑比例
- **护栏指标（避免只追次数）**
  - `Weekly Usage`、`WAEP`、`Reuse Rate` 不显著变差
- **判定标准（方向性）**
  - B 组少返工代理指标改善，且护栏指标健康

### Risk Mitigation

- **风险：约束包变成第二套模板系统，学习成本上升**
  - 缓解：默认推荐 + 高级折叠；MVP 内置约束包数量克制；场景化标签呈现。
- **风险：效果不显著或用户不理解**
  - **Fallback（降级路径）**：约束包退化为模板内 preset
    1) 主路径仅保留 1 个推荐 preset（不强调组合）
    2) 专家模式保留组合能力作为可选增强
    3) 遥测与校验继续保留，用数据验证 preset 是否足够
- **风险：过强约束让用户感觉被打断**
  - 缓解：除空内容/全占位符外均为非阻断提示；允许一键继续导出。

## Web App Specific Requirements

### Project-Type Overview

- **应用形态**：SPA（单页应用），以客户端路由组织主要页面与工作流。
- **SEO 策略**：不需要 SEO；因此 MVP 不引入 SSR/预渲染作为交付约束，重点投入交互效率与离线可靠性。
- **实时性**：不需要实时协作/实时同步/实时看板（避免引入 WebSocket/长连接等复杂度）。

### Platform Requirements

- **Browser Support Matrix（最低支持）**
  - Chrome：最新两版
  - Edge：最新两版
  - Safari：16+

### Responsive Design

- **响应式原则**：优先保障桌面浏览器的工作台式体验；在小屏上至少保证核心链路可完成（查看/编辑/复制或下载导出）。
- **输入方式**：以键鼠为主；触屏视为兼容场景（不为触屏设计专用交互）。

### Performance Targets

- **沿用 Step 3 的默认性能目标（体验可测）**
  - 生成到预览可用：p95 \(\le 300\)ms，p99 \(\le 800\)ms
  - 打开模板/切换约束包：p95 \(\le 200\)ms
  - 复制/导出动作完成：p95 \(\le 200\)ms

### SEO Strategy

- **不做项（MVP）**：SSR/预渲染、面向搜索引擎的页面收录优化。
- **仍需具备的基础**：清晰的页面标题/路由结构（用于用户收藏与恢复）、基础 meta 信息满足分享/可读性即可。

### Accessibility Level

- **目标等级**：基础可用。
- **最低要求**
  - 关键路径支持键盘操作（Tab/Shift+Tab、Enter、Esc），焦点可见。
  - 表单字段与错误/提示信息可被理解（语义化标签与可访问的提示文案）。
  - 颜色对比与字号满足基本可读性（避免仅靠颜色传达关键信息）。

## Project Scoping & Phased Development

### MVP Strategy & Philosophy

**MVP Approach:** Problem-Solving MVP（最小闭环解决核心问题，并做到可验收的稳定性/质量护栏）

**MVP Definition of “Useful”:**
- 用户在一次会话内完成：选模板 → 填变量 → 约束包（默认推荐）→ 预览 → 导出（复制/下载/全选兜底至少两条路径中任意一条完成即可）
- 导出链路满足 Step 3 约束：`export_error/copy_error` 目标为 0（SLO 周失败率 \(\le 0.01\%\)）；`export_denied` 必须兜底可完成导出
- 质量以“少返工代理指标（Valid Export Proxy）”作为护栏，不以“导出次数”单指标驱动

**Scope Guardrails (Phase 1):**
- 任何不直接提升“导出可完成性 / 离线可信 / 可验收口径”的能力，默认后移
- 本期无登录、无实时；任何强服务端依赖默认后移

**Resource Requirements (MVP):**
- 1–2 名前端/全栈（离线存储、导入导出、可观测与埋点、导出兜底）
- 0.5 设计/产品（信息架构、关键交互与文案、可用性验证）
- 可选：0.5 QA（关键链路回归：导出/权限拒绝/导入导出/数据恢复）

### MVP Feature Set (Phase 1)

**Core User Journeys Supported (MVP):**
- Journey 1（主路径）
- Journey 2（导出兜底）
- Journey 4（迁移/导入导出）
- Journey 5（隐私敏感）
- Journey 6（数据丢失/恢复）

> Journey 3（周骁）在 MVP 仅覆盖“最小可用”：允许创建模块并在模板中引用模块；不承诺复杂治理能力。

#### Must-Have Capabilities（不做就等于产品失败）
- **模板驱动主路径**：最近/收藏/搜索 → 表单化变量填写（未填高亮/跳转）→ 约束包默认推荐 + 高级折叠 → 预览 → 导出
- **导出动作面板（保证可完成）**
  - 复制成功 → `copy_success`
  - 下载 `.md/.txt` 成功 → `export_success`
  -（可选）可全选文本区兜底
- **导出前轻量校验**：默认非阻断；唯一硬拦截为空内容/全占位符（`validation_blocked`）
- **离线优先持久化**：模板/模块/约束包/最近组合/匿名标识本地保存
- **资产沉淀边界**：单次问题默认不入库；仅用户显式“保存为资产”才沉淀
- **迁移与恢复（离线可信底座）**
  - 资产包导入/导出（JSON），包含 `version`
  - 导入策略：合并/覆盖/新建副本
  - 版本兼容提示：不兼容时给出可理解的原因与建议路径
- **遥测最小化与可控（隐私信任门槛）**
  - 可关闭遥测、可重置匿名用户标识、可查看事件清单
  - 不上传完整 Prompt 原文（仅事件+必要摘要）
- **可验收事件与口径（MVP 就要可跑通）**
  - 事件：`export_success/copy_success`、`export_denied`、`export_error/copy_error`、`validation_blocked`、`export_duplicate`、`reuse_used`
  - 口径：`dedup_window_seconds=30`；周口径=用户本地时区 + 周一开始
  - 指标：Weekly Usage / WAEP / Reuse Rate / 少返工代理指标（护栏）

#### Can-Be-Manual Initially（可先低配，不阻塞 MVP）
- 内置模板/约束包扩充：先少量精选，后续扩展
- 导入安全识别：MVP 只做最小提示（摘要 + 可展开片段 + 打标复查），不追求高召回
- 可观测面板：MVP 先保证事件可采集可复现，报表/看板后置

#### Explicit Non-Goals（Phase 1 明确不做）
- 模型直连/对话托管
- SEO/SSR/预渲染
- 实时协作/实时同步
- 账号体系与多端同步
- 社区/市场/分享评分
- 团队协作与权限系统
- 复杂版本分支/对比/回滚（仅保留导入导出与基础 `version` 字段）

### MVP Release Gates（上线门槛：不满足就不算“可交付 MVP”）
- **导出可完成性（跨环境）**
  - 用户可通过导出动作面板完成导出：复制成功或下载成功（二选一即可）
  - 若剪贴板被拒绝（`export_denied`），必须仍能通过下载/可全选文本完成导出，且不丢失编辑状态
- **导出可靠性（零失败目标）**
  - `export_error/copy_error` 目标为 0（SLO 周失败率 \(\le 0.01\%\)）
  - `validation_blocked` 只用于空内容/全占位符硬拦截，避免误伤正常导出
- **离线可信（可迁移、可恢复）**
  - 资产包导出可用作“恢复锚点”（能导出、能导入）
  - 导入必须提供合并/覆盖/新建副本策略，并给出版本/兼容提示
- **可验收与可学习（数据闭环最低集）**
  - `dedup_window_seconds=30` 与按周口径在数据侧可复现
  - 关键事件最小集可用（见上）
  - 遥测默认最小化且可关闭，并可查看事件清单（不上传完整 Prompt 原文）

### Scope Boundary Tests（范围边界测试：防 Phase 1 偷偷膨胀）
- 需要服务端账号/同步才能成立吗？如果是 → Phase 2+
- 需要多人实时/长连接才能成立吗？如果是 → Phase 2+
- 需要复杂版本分支/对比/回滚才能成立吗？如果是 → Phase 2+
- 不能直接改善“导出可完成性/离线可信/可验收口径”吗？如果是 → Phase 2+

### Post-MVP Features

**Phase 2 (Growth):**
- 资产治理增强：更细粒度版本管理（对比/回滚）、更强组织能力
- 导入可信机制增强：来源/版本/变更对比、更强提示工作流
- 约束包生态扩展：推荐策略优化、场景化标签体系（保持默认推荐与折叠策略）

**Phase 3 (Expansion):**
- 账号体系与多端同步（可选）
- 社区/市场（发布、评分、衍生与版本演进）
- 团队协作（资产库治理、权限与审计）

### Risk Mitigation Strategy

**Technical Risks:**
- 离线数据风险：资产包导出作为恢复锚点；导入策略清晰 + 兼容提示
- 导出可靠性：`export_denied` 分类 + 导出面板兜底，保证价值链路不中断
- 约束包复杂度膨胀：数量克制 + 默认推荐；必要时按 Step 6 fallback 降级为模板 preset

**Market Risks:**
- 约束包是否减少返工：按 Step 6 A/B 尽早验证（少返工代理指标为核心，护栏不恶化）

**Resource Risks:**
- MVP 范围失控：坚持“闭环优先”，治理与生态扩展推到 Phase 2
